## Requirements

### Requirement 1: 拒绝概念堆砌 — 强制"名词驱动"产出

AI 助理在协助用户设计架构时，只要提及诸如"插件中心"、"解耦服务"、"事件中心"之类的高级概念，就必须同步产出带有数据约束的"证明契约"。

#### Scenario 1.1: AI 提出 EventBus 解耦方案
- **WHEN** AI 提出将模块间通信通过 EventBus 解耦时。
- **THEN** 必须在同一文档中定义：
  - 每种事件类型的 `@dataclass(frozen=True)` 载荷类
  - EventBus 的 `emit()` / `subscribe()` 方法签名
  - 是否区分前台同步路径和后台异步路径
  - 后台路径的消费策略（重试、死信等）
- **ELSE** 该设计视为"无效的纯文字抽象"，不可进入 tasks.md 阶段。

#### Scenario 1.2: AI 提出插件化架构
- **WHEN** AI 提出系统应支持插件扩展时。
- **THEN** 必须在同一文档中定义：
  - 插件基类的完整接口（`setup()` / `teardown()` 方法签名）
  - 插件元数据结构（`Manifest` 的字段定义）
  - 插件发现机制（编程式注册 / entry_points 自动发现 / 两者兼备）
  - 插件 Context 的权限边界（暴露哪些 API，限制哪些访问）

#### Scenario 1.3: AI 提出状态机管理
- **WHEN** AI 提出使用状态机管理生命周期时。
- **THEN** 必须定义：
  - 状态枚举的完整集合
  - 合法转换白名单 (`TRANSITIONS` dict)
  - 非法转换的错误处理策略（抛异常类型）

### Requirement 2: 端口契约强制化 — 防泄漏边界声明

一旦系统包含多层架构（核心层 + 外围层 / 框架 + 插件），设计文档必须硬性声明端口契约层。

#### Scenario 2.1: 定义系统对外操控接口
- **WHEN** 设计文档描述核心引擎对外暴露的操控方法时。
- **THEN** 必须以 `Protocol` 类声明接口契约：
  - 入参只允许基础类型（`str`, `int`, `dict`, `list`）
  - 返回值只允许 `dict` 或 `list[dict]`
  - 严禁直接暴露领域对象（如 `QWidget`, `StateMachine`, `Enum` 实例）
- **验证方法**: 检查 Protocol 类中是否存在任何非基础类型的参数注解。

#### Scenario 2.2: 底层引用的只读保护
- **WHEN** 设计中需要向受信任模块暴露底层引用时。
- **THEN** 必须使用只读 `@property` 暴露，类型标注使用 `Any` 避免强运行时依赖，严禁直接赋值属性暴露。

### Requirement 3: 多层权限沙盒 — 禁止"单层 Context"

任何涉及插件/扩展架构的设计，不得使用单个扁平 Context 同时服务所有调用者。

#### Scenario 3.1: 插件权限分级
- **WHEN** 系统同时存在第三方插件和内置核心插件时。
- **THEN** 必须设计至少两层 Context：
  - **普通 Context**: 只暴露注册型 API（事件订阅、UI 插槽注册、钩子注册、配置读取）
  - **高权 Context (AdminContext)**: 在普通基础上，额外通过只读 `@property` 暴露底层引擎引用
- **权限分配**: 通过白名单机制在编排器中决定哪些插件获得高权 Context。

### Requirement 4: 钩子系统精度 — 禁止降级为"try-catch"

当设计中包含钩子/拦截器机制时，不得将其简化为仅有异常捕获的 wrapper。

#### Scenario 4.1: 钩子系统设计
- **WHEN** 设计文档提出引入钩子系统来允许插件拦截/修改核心流程时。
- **THEN** 必须明确定义：
  - 执行策略枚举（至少包含 PARALLEL, WATERFALL, BAIL_VETO）
  - 每种策略的语义（并发执行 / 瀑布传导 / 一票否决）
  - 熔断器设计（失败阈值、超时控制、三态模型 CLOSED/OPEN/HALF_OPEN、恢复窗口）
  - 每个钩子的强类型载荷定义（区分 frozen 不可变 vs mutable 可变）

### Requirement 5: 检查点阻断 — tasks.md 的防暴机制

当 tasks.md 包含对核心底层（通信总线、跨线程、协程调度）的重构任务时，必须插入强制校验阻断点。

#### Scenario 5.1: 高危重构步骤
- **WHEN** tasks.md 中的任务涉及跨线程验证、网络协议或底层状态机实现时。
- **THEN** 必须在该任务组末尾插入 `【检查点】` 任务：
  - 要求向用户展示测试输出 / 运行 Log
  - 明确标注"必须获得用户显式确认后才能推进到下一阶段"
- **禁止**: 从基础设施任务直接跳到完整功能运行的"无阻断大连环"。

#### Scenario 5.2: 物理防渗透规定
- **WHEN** 核心逻辑层文件（如状态机、事件载荷）应当保持平台无关时。
- **THEN** 在对应 task 中加粗写入"`【防腐规定】该文件内严禁出现任何 [特定框架] 的 import`"。

### Requirement 6: 对标法 (Alignment Protocol) — 参照系统设计新模块时的 SOP

当 AI 被要求参考现有系统设计新模块时，必须遵循对标法标准操作流程。

#### Scenario 6.1: 用户要求参考系统 A 设计系统 B
- **WHEN** 用户指令包含"参考 X"、"对标 Y"、"复刻 Z 的架构"等字样时。
- **THEN** AI 必须：
  1. 列出参照系统中需要对标的**每一个源码文件**（不是概念文档）
  2. 逐文件提取核心类、方法签名、数据结构
  3. 建立「参照模块 → 新模块」的映射表，标注"原样复刻"或"需要适配"
  4. 在设计文档的每个 Decision 中标注对标来源文件和具体类名
  5. 完成设计后逐行对照映射表，标记覆盖情况（✅ / ❌ / ⚠️）
- **禁止**: 仅阅读概念性文档（如 architecture.md）后就产出设计，绕过源码级对照。

### Requirement 7: 正面经验保护 — Postmortem 必须包含"做对了什么"

任何 postmortem 文档不得只记录失败面。

#### Scenario 7.1: 编写 Postmortem
- **WHEN** 创建任何类型的 postmortem 或回顾文档时。
- **THEN** 必须包含独立的"正面经验"章节，明确列出：
  - 哪些做法是正确的、应被保留的
  - 哪些设计是原创性的好设计（即使整体方案有缺陷）
  - 哪些方向性判断是准确的
