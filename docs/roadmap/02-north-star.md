# 终局产品

这份文件回答一个问题：

**Flow 的终点不只是“功能更多”，而是什么样的完整且优雅的产品。**

这份定义已经吸收了 `docs/past/aim.md` 中仍然有效的终局产品方向，但所有内容都按“目标能力”重写，不再把愿景原文中的“Current Feature Set”口吻当成当前事实。

## 北极星定义

完整且优雅的 Flow，应该同时满足两个层面：

### 1. 用户层

用户在日常使用中会感受到：

- 任务录入和切换几乎没有摩擦。
- 系统始终把注意力压缩到一个当前任务。
- 恢复任务时，拿回的是工作现场，而不只是一个标题。
- HUD 像桌面空气一样存在，轻、准、克制，但有足够表达力。
- CLI、daemon、HUD、Windows 入口表达的是同一套产品语义。
- AI 是放大器，不是第二套复杂系统。

### 2. 系统层

系统内部应具备：

- 稳定且可验证的任务流主链。
- 产品级的上下文捕获、存储、恢复和轨迹系统。
- 可组合、可扩展、线程安全的 HUD runtime。
- 可降级、可替换 provider 的 AI 边界。
- 真正接入产品流的插件生态。
- 稳定可维护的跨端交付路径。

## “优雅”在这个项目里具体指什么

这里的“优雅”不是视觉修饰，而是几个更硬的标准：

### 1. 用户认知简单

- 系统不把内部复杂度暴露给用户。
- 一个命令、一个状态、一个界面，在不同入口下表达一致。
- 失败时给出的是可解释状态，而不是底层实现细节。

### 2. 交互克制

- HUD 不抢屏幕，不变成大而全控制台。
- AI 不劫持主流程，只在需要时提供压缩后的建议。
- 自动化不会破坏“单任务主协议”。

### 3. 恢复比记录更重要

Flow 不应只是记录你做过什么。
它的真正差异化能力，是帮助你回到之前正在做的那件事。

### 4. 扩展比重写更重要

后续新能力应优先通过既有边界接入：

- 后端 runtime
- 前端 runtime
- plugin context
- protocol contract

而不是不断新增旁路逻辑。

## 终局产品的六条主轴

### 1. Core Loop

任务创建、排序、启动、暂停、阻塞、恢复、完成，这条链必须永远稳定。

### 2. Context Recovery

任务切换要接近“工作现场切换”，而不是纯任务状态切换。

### 3. HUD Experience

HUD 必须从 MVP 卡片演进成真正的桌面心流界面。

### 4. AI Assistance

AI 应该承担两个角色：

- 把复杂任务压缩成可执行步骤
- 在当前上下文下给出下一跳建议

### 5. Plugin Platform

插件必须进入真实产品控制流，而不是只停留在漂亮抽象。

### 6. Delivery And Independence

Windows/WSL 路径要稳定，同时为未来完全协议解耦保留演进空间。

## 从 `docs/past/aim.md` 吸收后的七个能力目标

### 1. 单任务协议必须始终成立

终局产品不只是“支持任务状态”，而是要把下面这些东西做成稳定体验：

- 八维状态模型
- 单一 `In Progress` 排他约束
- 自动上下文切换
- 入口一致的状态语义

### 2. 上下文恢复必须接近“时光倒流”

真正要实现的不是简单 snapshot，而是：

- 系统级快照
- 显式挂载与隐式捕获并存
- 切回任务时恢复工作现场
- 逐步形成任务专属轨迹

### 3. HUD 必须是沉浸式而不是打扰式

`docs/past/aim.md` 中最有价值的 HUD 方向应继续保留：

- hover-to-interact
- 截止时间压迫感视觉化
- 柔性 Flowtime 提醒
- 轻、准、克制的桌面存在感

### 4. 数据层不只负责存储，还要负责可回看性

终局产品的数据层目标包括：

- Git 驱动账本
- 可解释的历史变更
- 零点击复盘材料
- 被动上下文轨迹

### 5. AI 只做高价值压缩

保留 `docs/past/aim.md` 中真正有价值的两点：

- 把复杂任务降维成可执行微步骤
- 在当前上下文和时间窗口下给出下一跳建议

### 6. 扩展面要同时服务第一方和第三方

终局产品应同时拥有：

- 能直接支撑产品体验的第一方插件
- 面向社区的第三方扩展 API

### 7. 外部入口和前沿接口应成为后期放大器

`docs/past/aim.md` 中这些方向可以保留，但它们属于后期放大器，而不是当前主线：

- 消息网关接入
- MCP server 化
- 本地 RAG 记忆胶囊
- 主动条件轮询

更细的能力拆解见：

- [09-capability-decomposition.md](./09-capability-decomposition.md)

## 不属于北极星的东西

下面这些方向应被明确排除：

- 为了架构洁癖而重做已经稳定的底盘
- 在产品主线没完成前，过早投入大规模协议仓提取
- 继续把 HUD 做成“会动的调试界面”
- 把 AI 做成绕开主产品逻辑的第二套系统
- 在 launcher 里堆产品逻辑

## 终局版本的最小判断标准

当下面这些条件同时成立时，才可以说 Flow 接近完整且优雅：

- 日用用户可以稳定依赖它管理和恢复工作流。
- 任务切换有真实上下文恢复能力。
- HUD 足够轻，但足够清楚地表达状态、压力和建议。
- AI 可以稳定拆解任务，并在失败时优雅降级。
- 至少存在一批真实可用的第一方插件。
- 新功能不再逼着系统回到“大泥球”结构。
