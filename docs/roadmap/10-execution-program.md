# 10. 执行总图

状态基线：2026-04-17

这份文件不是愿景文档，也不是现状文档。
它回答的是另一个问题：

**如果从今天的代码基线出发，怎样一步步推进，才能最终到达 `02-north-star.md` 里的产品目标。**

## 一. 这套执行文档要解决什么问题

现有 roadmap 已经回答了三件事：

- 现在到了哪里
- 终点是什么
- 大方向和先后顺序是什么

但还缺第四件事：

- 每一步具体怎么做，才能不跳步、不返工、不再次把愿景当现状

这套执行文档因此增加了五个强约束。

### 1. 必须从当前代码基线出发

每个阶段都只允许建立在 `01-current-state.md` 已经核实的能力之上。

### 2. 每一步都必须有明确产出物

允许的产出物包括：

- 新的或更新后的契约
- 明确的 runtime path
- 更清楚的 payload / config / data model
- 可执行验证
- 操作手册或交付脚本

### 3. 每一步都必须说明验证方法

如果某一步没有明确验证，它就不算真正完成。

### 4. 每一步都必须说明前置条件

任何依赖上一阶段语义的工作，都不允许抢跑。

### 5. 每一步都必须说明“现在不做什么”

这样可以防止愿景不断外溢，把路线重新拉回大泥球。

## 二. 从现状到终局的六道阶段门

整条路线分成六道连续的阶段门。
后一道门默认建立在前一道门已经通过的前提上。

```text
当前代码基线
  -> Gate A: 单机日用闭环
  -> Gate B: 上下文恢复成形
  -> Gate C: HUD 产品化
  -> Gate D: AI 增强接入
  -> Gate E: 插件与交付稳固
  -> Gate F: 外围入口与前沿能力
  -> 北极星产品
```

## 三. 六道阶段门的真实含义

### Gate A：单机日用闭环

通过标准：

- 后端主任务流足够稳定
- CLI / daemon / HUD 的任务语义一致
- 本地 launcher / 启动路径可重复验证
- 默认错误路径可以被解释

当前状态：

- Gate A 的 repo-owned day-use contract、baseline、runbook 和 smoke gate 已经发布到 `docs/day-use/`。
- 对应的实现与验证证据已经完成并通过 challenger 关闭。

如果 Gate A 没过，后面所有能力都会建立在“不好用但能跑”的底座上。
如果 Gate A 已经过了，后续工作就应该以这些日用文档和验证结果作为稳定前置，而不是重新发明一套 Gate A 语义。

### Gate B：上下文恢复成形

通过标准：

- snapshot 数据模型不再过于贫瘠
- capture / restore 语义明确
- 显式挂载与隐式捕获边界明确
- 恢复失败时有清楚降级路径

如果 Gate B 没过，HUD 只能展示状态，AI 也拿不到足够有价值的上下文。

### Gate C：HUD 产品化

通过标准：

- HUD 不再只是 task-status MVP
- 交互、状态表达、提醒能力已开始形成产品层
- 所有增强仍然挂在现有 runtime 上

如果 Gate C 没过，Flow 无法形成明显的桌面产品感。

### Gate D：AI 增强接入

通过标准：

- `flow breakdown` 已成为真实能力
- provider boundary 清楚
- AI 失败不会污染主任务流

如果 Gate D 没过，AI 仍只是想法，不是可交付能力。

### Gate E：插件与交付稳固

通过标准：

- 第一方参考插件已经开始形成
- launcher / 交付规则制度化
- 需要长期治理的边界有明确触发条件和退出路径

如果 Gate E 没过，系统扩展和交付会持续依赖临时经验。

### Gate F：外围入口与前沿能力

通过标准：

- 消息网关、自动复盘、MCP / RAG / 主动轮询这类能力不再是孤立实验
- 它们复用已经稳定的 task-flow / context / HUD / provider 边界

如果 Gate F 没过，前沿能力只会增加复杂度，而不会形成护城河。

## 四. 整条路线的禁止跳步规则

### 1. 不允许在 Gate A 之前大规模做 AI

原因：

- 主任务流和操作路径如果还不稳定，AI 只会把底层问题掩盖掉。

### 2. 不允许在 Gate B 之前把 HUD 做成复杂交互台

原因：

- 没有足够好的上下文语义，HUD 再精致也只是状态展示器。

### 3. 不允许在 Gate C 之前过早讲开放生态

原因：

- 前端产品流本身都没成熟，讲第三方生态只会过度承诺。

### 4. 不允许在 Gate D 之前扩张 provider 范围

原因：

- 第一个真实问题都没解决，多 provider 只会扩大调试面。

### 5. 不允许在 Gate E 之前把机器特定 launcher 当成通用分发

原因：

- 当前 launcher 是现实样本，不是最终答案。

## 五. 每个阶段都必须同时回答的五个问题

后面的 11-16 号文档都会按同一个模板展开。

### 1. 这个阶段的目标是什么

不写愿景大词，只写阶段性能力目标。

### 2. 这个阶段建立在什么前置上

不满足前置，后面步骤默认不能开始。

### 3. 这个阶段具体拆成哪些连续步骤

每一步都必须能单独完成、单独验证。

### 4. 每一步的验证是什么

至少写清：

- 需要看哪些现有测试
- 需要新增哪些验证
- 需要哪些真实运行验证

### 5. 这个阶段完成的门槛是什么

没有阶段门槛，就没有办法判断“该不该进入下一阶段”。
